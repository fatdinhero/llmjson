### TODO-004: Linux-Support via llama-cpp-python
- **What:** Backend-Abstraktion einführen, llama-cpp-python als zweites Backend
- **Why:** Erweitert Zielgruppe auf Linux-Server. Erst wenn Nachfrage validiert (GitHub Issues).
- **Pros:** Viel größerer adressierbarer Markt, CUDA-GPU-Support
- **Cons:** Backend-Abstraktion-Design, neue Dependency, eigene Tests. Logit-API von llama-cpp-python muss verifiziert werden.
- **Context:** llama-cpp-python bietet `logits` Property auf LlamaState. Machbarkeitstest: 4h.
- **Depends on:** Mindestens 3 GitHub Issues die Linux-Support anfragen

### TODO-005: MRC als opt-in Feature
- **What:** Soft-Resonanz-Layer zurück ins Package als `generate(..., use_mrc=True)`
- **Why:** Verbessert Inhaltsqualität (nicht Korrektheit). Könnte bei sinnvolleren Werten in JSON-Values helfen.
- **Pros:** Zeigt volle OGJ-CF Architektur, differenziert von reinem Grammar-Constraining
- **Cons:** +1200 Zeilen Code, Schema-Encoder als Dependency, höhere Komplexität
- **Context:** MRC/WMS/QAS wurden in D9 aus dem Prototyp ausgeschlossen. Wieder einbauen nur wenn Content-Qualität ohne MRC unbefriedigend ist.
- **Depends on:** Nutzerfeedback zur Content-Qualität

- **Cons:** Muss pro-Modellfamilie getestet werden
